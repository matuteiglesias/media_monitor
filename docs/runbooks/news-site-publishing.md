# News site publishing audit and strategy

## Root-cause summary

The current publish ladder has two truths:

| Layer | Expected role | Audit result | Broken seam |
| --- | --- | --- | --- |
| `storage/indexes/*` | Runtime truth generated by index builders | The news index builder can emit empty `news_recent_*_latest.jsonl` files when no source buses are available. | Validation accepted empty news surfaces, so a publish could look successful while producing no public news. |
| `apps/news_site/public/data/*` | Deploy snapshot consumed by the Next app | The committed news snapshots were still at `digest_at=20260506T15`, while the editorial snapshot had a newer fallback/degraded timestamp. | Public snapshots can remain stale unless the refresh step runs and the results are deployed. |
| `vercel.json` | Production route/cache contract | Root config still referenced legacy `/web/data/*` and rewrote `/` to `/web/index.html`. | The cache rules did not match the Next app's real `/data/*` public path, and the rewrite made the production root ambiguous. |
| Next app loaders | Render surface | Loaders read files from `apps/news_site/public/data` at build/runtime via the app working directory. | This part is correct, but it bakes whatever public snapshot is present during `next build`. |
| Editorial fallback | Emergency local preview only | Refresh silently wrote a plausible fallback when the canonical editorial index was absent. | Missing canonical editorial data could be hidden in deployed snapshots. |

## Durable near-term publish path

Use a single command from the repository root:

```bash
make publish-news-site DIGEST_AT=$(date -u +%Y%m%dT%H)
```

That command:

1. builds news access indexes,
2. builds editorial access indexes,
3. validates `storage/indexes`,
4. refreshes `apps/news_site/public/data`,
5. smoke-tests public snapshots against storage snapshots,
6. builds the Next app, and
7. writes `apps/news_site/public/data/publish_manifest.json` with the build SHA and validation digests.

Deploy only after this command passes. For production, do not set `ALLOW_EDITORIAL_FALLBACK=1`; that flag is reserved for local emergency previews and makes smoke checks accept a fallback editorial surface.

## Cache and deployment contract

Production should deploy the `apps/news_site` Next app, where static data lives at `/data/*` and the debug endpoint lives at `/api/debug-data`. However, the current Vercel project root was not verified in this audit. Because production may still depend on the legacy root `/web/index.html` rewrite, route/cache changes should be isolated in a separate deployment PR after confirming the Vercel project root and live routes.

## Publishing strategy options

### Option A — Keep committing snapshots

Pros:
- simple static deployment,
- immutable data history,
- no new database or bucket.

Cons:
- noisy git history,
- build/deploy latency for content-only updates,
- source changes and data refreshes are mixed,
- stale generated artifacts are easy to commit by accident.

### Option B — Keep app static, publish latest data to object storage

Examples: Vercel Blob, S3/R2, Supabase Storage, GitHub Pages/raw artifact, or another signed JSON endpoint.

Pros:
- no commit spam for every news refresh,
- app can fetch current JSON with explicit cache/revalidation,
- source-code deploys are separate from content refreshes.

Cons:
- needs credentials and bucket lifecycle management,
- publisher must validate and atomically promote snapshots,
- requires stricter data endpoint security and observability.

### Option C — Next API route pulls latest data at request time

Pros:
- no committed snapshots,
- freshness can be controlled at request/revalidation time.

Cons:
- serverless runtime needs access to canonical data,
- more cache/revalidation complexity,
- production incidents become harder to reason about than static files.

### Option D — Scheduled GitHub Actions builds/deploys snapshots

Pros:
- automation is explicit,
- repository can remain the operational source of truth.

Cons:
- still creates commit/deploy churn if snapshots are committed,
- artifact-only deploys need a separate release discipline.

## Recommendation

Near term, keep static snapshots but require `make publish-news-site` before every deploy. This keeps the current architecture while making missing, empty, invalid, stale, or fallback snapshots fail loudly.

Medium term, stop committing every news refresh. Publish validated JSON/JSONL snapshots to external object storage or an artifact endpoint, and let the Next app read that endpoint with explicit cache headers or revalidation. Keep source-code commits/deploys for application changes, not hourly content churn.
